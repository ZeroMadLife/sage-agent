## 代码质量评估

### ✅ 做得好的地方

**1. LLM 输出归一化层 — 这是整个 Phase 2 最有价值的设计**

`agents/planning.py` 实现了一套完整的 `_normalize_itinerary_data()` 归一化层，处理 LLM 输出的各种 schema 漂移：

```python
# LLM 可能返回：
# - ticket_price: "30" (字符串) → _to_int → 30
# - spots: [{"id": "x", "name": "西湖"}] → spot_id 回退到 name
# - meals: "午餐" (字符串) → [{"name": "午餐", "meal_type": "other", ...}]
# - transport: "出租车" (字符串) → [{from_name: "行程交通", to_name: "出租车", ...}]
```

这是**生产级思维**——不盲目信任 LLM 输出，在 Pydantic 校验前先做容错归一化。面试时这是强加分点，说明你理解 LLM 输出的不确定性。

**2. 图拓扑用 barrier edge 正确实现并行**

```python
# graph.py:42 — 关键设计
graph.add_edge(["info", "recommend"], "planning")
```

LangGraph 的 barrier edge 语法 `add_edge([node1, node2], target)` 确保 info 和 recommend 两个并行节点**都完成后**才执行 planning。这比计划中写的两条独立边（`info→planning` + `recommend→planning`）更简洁正确。

**3. 预算降配循环的 e2e 测试验证了真实迭代**

```python
# test_e2e.py:143 — 验证超支重规划
planning_llm.ainvoke = AsyncMock(
    side_effect=[
        MagicMock(content=_json_content(total_cost=600)),  # 第一次：超支
        MagicMock(content=_json_content(total_cost=300)),  # 第二次：预算内
    ]
)
# 验证 LLM 被调用了2次，最终在预算内
assert planning_llm.ainvoke.await_count == 2
assert result["budget_breakdown"].over_budget is False
```

这个测试不只是验证 happy path，而是**验证了降配循环真的会触发并收敛**——这是 Multi-Agent 协作的核心证据。

**4. info_node 在 graph 中做了适配 — 只暴露 weather_info**

```python
# graph.py:29-32 — 避免并行节点写同一字段冲突
async def info_weather_node(state):
    result = await info_agent(state)
    return {"weather_info": result.get("weather_info", {})}
    # 注意：不返回 recommendations，因为 recommend 节点也会写这个字段
```

info_node 和 recommend_node 都会写 `recommendations` 字段。在并行执行时如果两个都写，后者会覆盖前者。Codex 在 graph 层做了一个适配器只取 weather_info，避免了并行写入冲突。这是对 LangGraph 状态合并机制的深入理解。

**5. 天气优雅降级 — 核心链路不依赖外部服务**

```python
# info.py:34-41 — 天气失败不阻断行程生成
except Exception as exc:
    weather_info = {"current": None, "forecast": [], "error": True, "message": ...}
```

天气是增强信息，不是核心依赖。查询失败时标记 error=True，规划Agent 的 prompt 中会显示"天气信息不可用"，行程仍可生成。这是正确的容错设计。

**6. LLM 工厂用 SecretStr 包装 API key**

```python
# core/llm.py:84
return ChatOpenAI(api_key=SecretStr(api_key), ...)
```

Pydantic 的 SecretStr 在 __repr__ 中会遮蔽 key 值（显示 `**********`），防止日志/调试时泄露密钥。计划没要求这个，是 Codex 主动加的。

### ⚠️ 建议优化项（非阻塞）

**1. budget_node 的 LLM 参数未使用**

```python
# budget.py:45
_ = llm  # 显式标记未使用
```

当前预算校验是纯算法（比较 spent vs total），不需要 LLM。但 `create_budget_agent(llm)` 仍接收 LLM 参数。这是为 Phase 4 的"确定性验证器"预留的——到时预算降配建议需要 LLM 生成。

**建议：** 保留现状，加个注释说明这是为 Phase 4 预留即可。当前已有 `_ = llm` 标记。

**2. recommend_node 和 info_node 都搜索景点 — 重复调用**

info_node 和 recommend_node 在阶段1并行执行，都调用了 `scenic_client.search_scenic_spots()`，搜索参数几乎相同（都是 city + keywords from preferences）。

**影响：** 实际场景下 ScenicClient 是本地 JSON 文件（无网络开销），重复调用成本可忽略。但迁移到 PostgreSQL 后会有重复查询。

**建议：** Phase 3 优化——recommend_node 可以只做排序，景点数据由 info_node 统一搜索后在 state 中共享。但当前 MVP 不阻塞。

**3. demo_chat.py 的 parse_input 是硬编码关键词匹配**

```python
pref_keywords = {
    "美食": ["美食", "吃", "小吃"],
    "自然风光": ["自然", "山水", "湖", "风景"],
    ...
}
```

这是 MVP 阶段的简化方案。Phase 2 后续或 Phase 4 应该用 LLM 做意图识别替换硬编码匹配。

**建议：** 当前可接受。面试时说明"MVP用关键词匹配快速验证，后续会换成LLM意图识别"。
