## 代码质量评估

### ✅ 做得好的地方

**1. `_maybe_await` — 同步/异步 Redis 兼容**

```python
# core/memory/short_term.py:15
async def _maybe_await(value: Any) -> Any:
    if inspect.isawaitable(value):
        return await value
    return value
```

这个设计很聪明 — `fakeredis`（测试用）是同步的，`redis-py`（生产用）是异步的。`_maybe_await` 让同一套代码兼容两种客户端，测试用 fakeredis 不需要 async 包装。计划里没要求这个，是 Codex 主动解决的工程问题。

**2. Mem0 SDK 版本兼容 — `_unwrap_results`**

```python
# core/memory/long_term.py:67
@staticmethod
def _unwrap_results(raw_results: Any) -> list[Any]:
    if isinstance(raw_results, list):
        return raw_results
    if isinstance(raw_results, dict):
        results = raw_results.get("results")
        if isinstance(results, list):
            return results
        memories = raw_results.get("memories")
        if isinstance(memories, list):
            return memories
    return []
```

Mem0 SDK 不同版本返回格式不同（有的返回 list，有的返回 `{"results": [...]}`，有的返回 `{"memories": [...]}`）。这个方法统一适配了三种格式，避免 SDK 升级后 break。这是生产级思维。

**3. Mem0 工厂的 embedder 多方案支持**

```python
# core/memory/mem0_factory.py
# 支持 HuggingFace（本地，免费）和 OpenAI（远程，需key）两种 embedder
# 自动根据 provider 选择 model 和 dims
# 预置了 bge-large-zh/bge-small-zh/text-embedding-3-small/3-large 的维度映射
```

计划只要求 HuggingFace embedder，但 Codex 额外加了 OpenAI embedder 的备选方案和维度自动推断。这在 HuggingFace 模型下载失败时能快速切换到 OpenAI embedding。

**4. 三层优雅降级 — 记忆系统完全不阻断核心流程**

每一层都有 try/except 降级：

```
Mem0 初始化失败 → create_mem0_client() 返回 None
  → graph.py: memory_manager=None → 不插入 memory_node → 走 Phase 2 原有路径

Mem0 搜索失败 → LongTermMemory.search() 返回 []
  → memory_node: memory_context="" → planning 不注入记忆，正常生成行程

Mem0 存储失败 → LongTermMemory.extract_and_store() 静默跳过
  → 不影响用户响应

LLM 摘要失败 → ShortTermMemory.summarize() 返回原始消息
  → 不压缩但也不丢数据
```

这意味着即使 Docker 没启动、Qdrant 挂了、Mem0 SDK 报错，系统仍然能正常生成行程。这是正确的容错设计。

**5. graph.py 向后兼容 — `memory_manager=None` 时不插入记忆节点**

```python
# agents/graph.py:41-50
if memory_manager is not None:
    graph.add_node("memory", create_memory_node(memory_manager))

if memory_manager is not None:
    graph.add_edge(["info", "recommend"], "memory")
    graph.add_edge("memory", "planning")
else:
    graph.add_edge(["info", "recommend"], "planning")
```

Phase 2 的 e2e 测试不需要传 memory_manager，走 else 分支，原有路径不受影响。这保证了 Phase 2 的 94 个测试零回归。

**6. 摘要压缩的 prompt 设计**

```python
summary_prompt = (
    "请将以下对话历史压缩为一条简洁摘要，保留用户偏好、预算、目的地、"
    "已确认行程参数和重要决策。\n\n"
    f"{conversation_text}\n\n只输出摘要内容。"
)
```

摘要 prompt 明确指定了要保留什么（偏好/预算/目的地/行程参数/决策），不是泛泛的"摘要一下"。这保证了压缩后关键信息不丢失。

### ⚠️ 建议优化项（非阻塞）

**1. `extract_and_store` 是同步调用但标了 async**

```python
# core/memory/long_term.py:51
async def extract_and_store(self, conversation: str) -> None:
    try:
        self._mem0.add(conversation, user_id=self._user_id)  # ← 同步调用
```

Mem0 的 `add()` 方法是同步的（内部会调 LLM + 向量存储），但这里标了 `async`。在 asyncio 事件循环中会阻塞。不过当前 MVP 阶段记忆提取是在 Agent 回复后调用的，不阻塞用户响应，影响可接受。

**建议：** Phase 4 可以用 `asyncio.to_thread()` 包装：
```python
await asyncio.to_thread(self._mem0.add, conversation, user_id=self._user_id)
```

**2. `demo_memory.py` 没有清理之前的数据**

如果多次运行 demo_memory.py，Mem0 会累积重复记忆（"用户喜欢海鲜"存多份）。Mem0 内部有去重机制（ADD/UPDATE/NOOP），但多次运行可能产生近似重复。

**建议：** demo 开始前加一个 `mem0_client.delete_all(user_id=user_id)` 清理旧数据。非阻塞，不影响功能。

**3. `ShortTermMemory` 还没集成到 graph 的 Checkpointer**

当前 `ShortTermMemory` 是独立组件，还没接入 LangGraph 的 Checkpointer 机制。graph 执行过程中的状态持久化（中断恢复）还没实现。

**影响：** 如果服务器在 graph 执行中途重启，状态会丢失。但 MVP 阶段 graph 执行时间 <10 秒，这个风险极低。

**建议：** Phase 4 或 Phase 5 接入 `RedisSaver` 作为 LangGraph Checkpointer。
