# Sage README 与持续学习手册设计

> 日期：2026-07-20
>
> 基线：`dev/sage-v7` / `23a0090`
>
> 状态：已确认，进入实现
>
> 范围：信息架构、视觉资产、文档维护规则与验收标准；不包含本轮实现

## 1. 背景

Sage 已经具备产品主页、Assistant、Practice、Knowledge、公开成长门面和较完整的
Harness/Knowledge 实现，但仓库对外表达仍存在三类问题：

1. 根 README 以文字和能力清单为主，没有用真实产品界面建立第一印象。
2. `release/v7-beta/learning/` 已累积 00–14 共 15 个文件，部分内容带有面试备忘、
   绝对路径、易过期计数和未经源码复核的竞品结论，尚未形成可持续维护的学习手册。
3. 产品文档、发布证据、学习笔记和个人成长叙事的职责混在一起，后续容易继续堆叠。

当前发布材料存在分支分叉：`main@b30c2ad` 通过 PR #52 保存了
`release/v7-beta/README.md`、`CHANGELOG.md`、`REVIEW.md`、`TESTING.md` 和早期
`00-reading-map.md`；`dev/sage-v7` 则只有后续扩展的 learning 00–14，没有四个版本入口。
实现阶段应把 `main` 的四个文件当作待审计素材、把 `dev/sage-v7` 的当前代码和测试当作
行为事实，在开发分支合成统一版本，不能直接 cherry-pick 旧材料或只按旧汇报重建。

## 2. 目标与非目标

### 2.1 目标

- 把根 README 写成成熟开源项目主页：先说明 Sage 是什么、展示真实产品，再提供架构、
  快速开始、边界和文档入口。
- 让 README 对潜在用户、贡献者和技术评审者首先成立，同时自然地让面试官看懂产品与
  工程质量，不写成求职页或玩具项目说明。
- 用一张产品概念图建立 Sage 的视觉识别，再用真实截图证明已经实现的体验。
- 把 `release/v7-beta/` 建成版本入口，把 `learning/` 建成会随代码持续修订的学习手册。
- 把公开成长门面作为独立的个人叙事表面；存在稳定公网 URL 后，由 README 提供入口。
- 所有能力、状态、计数和架构描述都能追溯到当前源码、测试或明确标注的计划文档。

### 2.2 非目标

- 不把 README 改成简历、博客首页或面试题合集。
- 不在本阶段实现公网 Harness、正式博客系统、评论系统或发布后台。
- 不把 AI 生成概念图当作真实 UI、架构证据或功能承诺。
- 不为每次修复新增一篇 learning 文档，也不保留按日期无限追加的流水账。
- 不在文档工作中顺带重构产品代码。

## 3. 三个对外表面的职责

| 表面 | 主要读者 | 回答的问题 | 不承担的职责 |
| --- | --- | --- | --- |
| 根 `README.md` | 用户、贡献者、技术评审者 | Sage 是什么、现在能做什么、如何运行与验证 | 个人成长长文、版本内每项变更 |
| `release/v7-beta/` | 试用者、评审者、维护者 | 这一版本交付了什么、如何验证、风险在哪里 | 重复根 README、复制全部源码解释 |
| `/#/public` 公开成长门面 | 对个人过程感兴趣的访客 | 为什么做、如何成长、有哪些真实作品与公开笔记 | 仓库安装手册、未接入的公网 Agent |

这三者互相链接，但不复制同一份内容。README 始终以 Sage 产品为主；公开页承担个人
定位、成长轨迹和博客式导航。

## 4. 根 README 信息架构

README 采用“产品先行、证据跟随、运行收口”的顺序。

### 4.1 首屏

内容顺序：

1. Sage 名称与一句话定位：`Personal AI Learning Companion`。
2. 一句中文价值描述，强调目标、个人知识、实践和可验证成长。
3. CI、运行时、前后端技术与 License 徽章；只保留会自动更新或长期稳定的徽章。
4. 一张横向产品概念图“知识生长工作台”。
5. 三个主入口：快速开始、产品界面、学习手册。
6. 稳定公网地址可用后，再增加“公开成长记录”入口。

概念图表现人、目标、知识来源、Sage 工作台、Practice 和 Evidence 之间的关系。图内不放
小字号技术文本，不表现虚构界面，不替代后续真实截图与架构图。

### 4.2 产品叙事

用短段落回答三个问题：

- Sage 为什么存在：把聊天、个人知识和真实实践连接成可恢复的长期学习过程。
- 与普通聊天界面的区别：对话负责行动，Knowledge 负责观察，Practice 负责验证，
  Timeline/Evidence 负责可追溯。
- 当前使用边界：本地优先、适合受控试用，生产部署与公网 Harness 仍需正式发布门禁。

### 4.3 真实界面

展示三张当前产品截图，顺序固定：

1. Assistant 主对话：目标、回答、阶段或运行证据同屏。
2. Knowledge：真实图谱或来源/Wiki 工作台，展示引用与结构关系。
3. Practice：工具、Diff、审批或测试结果，证明“理解进入实践”。

每张图只配 1–2 句说明，不做卡片堆叠。截图必须来自当前代码和演示数据，不出现本机
绝对路径、真实 token、邮箱、用户 ID、私有仓库名或未公开内容。

公开成长门面可以在“项目与成长记录”处用一张较小预览图或文字入口呈现，但不加入三张
核心产品截图，以免 README 从产品页变成个人主页。

### 4.4 能力、架构与运行

- “现在能做什么”按 Assistant、Knowledge、Practice、Harness 四组描述，避免按内部包枚举。
- 架构图使用 Mermaid 或受版本控制的 SVG，作为技术事实来源；AI 概念图不承担该职责。
- 快速开始只保留能在干净环境复现的最短路径，详细变量和排障链接到
  `docs/GETTING-STARTED.md`。
- 单列“当前边界”，明确本地/公网、静态公开问答/受限 Harness、开发迁移/生产迁移的差异。
- Loop 已人工关闭，README 不再把自动巡检与自动合并描述为正在运行的产品能力；历史设计
  仍可留在专门文档中。
- 结尾提供文档导航、贡献方式、分支策略和 License。

## 6. Release V7 Beta 信息架构

实现阶段在 `release/v7-beta/` 建立四个入口文件：

| 文件 | 唯一职责 | 允许包含 | 不应包含 |
| --- | --- | --- | --- |
| `README.md` | 版本导航与试用入口 | 版本定位、阅读顺序、硬阻塞、链接 | 完整能力重复清单 |
| `CHANGELOG.md` | 相对上一基线的变更 | 新增、修复、破坏性变更、已知限制 | 架构教程、动态测试总数 |
| `REVIEW.md` | 架构与风险评审包 | 边界、数据流、信任模型、未关闭风险 | 使用教程、宣传文案 |
| `TESTING.md` | 可执行验证与证据 | 命令、场景、环境、结果记录方式 | 无来源的“全绿”结论 |

动态测试数量和某次 CI 结果只写入 `TESTING.md` 的带日期验证记录；README 和 learning
引用测试入口，不复制易过期数字。

## 7. Learning 持续维护规则

### 7.1 现有目录不继续按流水账增长

00–14 保持为稳定主题。实现阶段逐章审查并原位修改：

- 删除本机绝对路径、求职口吻和只对作者有意义的临时说明。
- 把“某竞品没有某能力”改为有来源、有限定条件的对比，或直接删除。
- 把设计意图和已实现行为分开；源码与测试是当前行为的最终依据。
- 只保留能帮助理解系统边界、调用链、取舍和失败模式的内容。
- 框架原理引用官方文档或当前依赖版本，不凭记忆写实现细节。

只有出现一个现有章节无法容纳的顶层子系统时，才新增章节。一次修复、一次重构或一次
学习心得不构成新增章节理由。

### 7.2 统一章节模板

每章按以下结构维护；不适用的小节可省略：

1. 本章结论与适用边界。
2. 当前状态：已实现、部分实现、规划中或复盘风险。
3. 第一源码入口与一条主调用链。
4. 状态/数据落点与生命周期。
5. 为什么这样设计，以及考虑过的替代方案。
6. 失败模式、安全边界和不可做的事。
7. 对应测试、可执行验证或观测证据。
8. 自测问题与进一步阅读。
9. `Last verified against`：最后核验的 source commit。

### 7.3 更新 SOP

每次影响某个主题的功能小版本收口时：

1. 判断行为是否改变了现有章节的事实、边界、调用链或风险。
2. 若改变，直接修改对应章节，不在文末追加日期日志。
3. 更新 `Last verified against`，同时保留能复跑的测试入口。
4. 由 `00-reading-map.md` 检查导航和当前阶段，不复制各章正文。
5. 运行链接检查、Markdown 格式检查和 `git diff --check`。
6. PR 正文列明“修改了哪些学习结论”，与产品代码验证一起审查。

## 8. 公开成长门面接入规则

当前 `/#/public` 已形成合理的公开表面：产品项目、阶段闭环、真实 Knowledge 截面、成长
轨迹和限定资料静态问答相互一致。真实检查已覆盖阶段切换、问答提交、`390x844` 无横向
溢出和控制台无异常。

README 的接入分两步：

1. 未部署阶段：只在开发文档注明 `/#/public` 是本地预览路由，不出现
   `http://127.0.0.1:*` 的公开徽章或 CTA。
2. 稳定部署后：README 增加“公开成长记录”链接，目标必须是 HTTPS 稳定 URL；上线前验证
   手机访问、匿名权限、静态资源、GitHub 外链和公开资料边界。

公开问答继续明确标为静态资料问答，直到受限公网 Harness 具备独立账号、只读工具集、
速率限制、内容边界和审计记录后，才能更改描述。

## 9. 实施切片

1. **事实审计**：逐条对照 README、00–14 learning、`main` 的四个 release 入口、当前源码
   和测试，建立保留/改写/删除表。
2. **Release 入口**：把 `main` 的四个文件移植为待审计素材，在 `dev/sage-v7` 基线上改成
   职责单一的入口并链接现有 learning；不直接 cherry-pick 旧提交。
3. **真实截图**：准备演示数据，捕获 Assistant、Knowledge、Practice，完成桌面和移动验收。
4. **概念主视觉**：生成并筛选“知识生长工作台”，压缩并提交资产。
5. **README 重写**：按产品先行结构落地，接入真实截图、架构、快速开始和真实边界。
6. **Learning 就地校正**：先处理 00、01、02、06、09、11、12、14，再覆盖其余章节。
7. **整体校验**：链接、图片、命令、测试入口、事实一致性、`git diff --check` 与 GitHub 渲染。

每个切片使用职责清晰的小提交；不把所有文档和图片压成一个无法审查的大提交。

## 10. 验收标准

- README 首屏在 GitHub 上能直接识别 Sage 的名称、产品定位、当前状态和主视觉。
- README 读起来像正式开源产品，不出现“玩具项目”“求职作品”式自我评价。
- Assistant、Knowledge、Practice 三张截图都来自当前产品、无隐私信息且链接有效。
- 技术架构由 Mermaid/SVG 与源码说明支撑，不以 AI 概念图作为证据。
- README 不把已暂停的 Loop、公网 Harness 或未完成部署描述为可用能力。
- `release/v7-beta/` 四个入口职责互斥，读者可以从版本入口进入验证或深度学习。
- learning 章节不使用本机绝对路径，不做无证据的竞品断言，不重复累计历史结论。
- 公开成长门面未上线时不暴露 localhost 链接；上线后链接为可匿名访问的稳定 HTTPS URL。
- 所有 Markdown 相对链接与图片在仓库视角可解析，命令与测试入口可执行。
- 完成与改动匹配的测试、生产构建、`git diff --check` 和阶段收口回顾。

## 11. 风险与控制

| 风险 | 控制方式 |
| --- | --- |
| 文档领先于代码 | 每条能力绑定源码/测试入口；规划项显式标注 |
| 截图快速过期 | 固定演示数据与尺寸，结构变化时原位替换 |
| 概念图造成错误承诺 | 图注说明其为产品概念；真实界面紧随其后 |
| Learning 越写越长 | 稳定主题、原位修改、新章门槛、去重评审 |
| 公开页与 README 抢叙事 | README 讲产品，公开页讲个人过程，只做链接 |
| `main` 与 `dev` 的 Release 继续分叉 | 先在 `dev` 合成统一版本并验证 Git tree；完整发布门禁后再同步 `main` |
